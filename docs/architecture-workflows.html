BigBright ERP Backend – Architecture & Accounting Workflows

Runtime stack on top shows the actual Spring Boot app (multi-company security, controllers, orchestrator, outbox, Postgres, RabbitMQ). The lower section zooms into how accounting touches every portal: React admins create journals/reports, dealers fetch invoices, orchestrator jobs (sales/factory/payroll) auto-post ledgers, and everything lands in the GL tables and events stream.

Order → Invoice Flow (What Actually Runs)

Dealers hit POST /api/v1/sales/orders in SalesController (erp-domain/.../sales/controller/SalesController.java). The controller hands off to SalesService.createOrder, which assembles the aggregate, assigns an order number, persists it, and publishes SalesOrderCreatedEvent. Once the transaction commits, OrderAutoApprovalListener invokes CommandDispatcher.autoApproveOrder, which calls IntegrationCoordinator.autoApproveOrder directly in the same HTTP thread—no queue, outbox, or retry envelope.

  1. Reserve inventory: IntegrationCoordinator asks inventory to reserve finished goods. Failures bubble up immediately; nothing retries.
  2. Journal booking: createAccountingEntry posts Dr AR / Cr Revenue via AccountingService.createJournalEntry.
  3. Dispatch & COGS: postCogsEntry runs once slips mark dispatched, again calling AccountingService.
  4. Invoice materialisation: InvoiceService.issueInvoiceForOrder persists Invoice + lines and bumps Dealer.outstandingBalance.
  5. Status update: Order status flips to APPROVED / INVOICED before the HTTP response returns.

The journal helper (postJournal) simply picks the first two accounts it finds (creating “Auto Cash/Auto Revenue” placeholders when needed) and pushes a one-sided debit/credit pair with no dealer linkage or invoice reference beyond SALE-{orderId} / COGS-{orderId}. Because the coordinator ignores the companyId parameter and relies on the servlet filter to keep thread-local context, the postings are only correct when invoked from the web tier.

Dealer Ledger Reality

InvoiceService updates a denormalised Dealer.outstandingBalance column and that value flows straight through Dealer APIs and reports (DealerService.listDealers, ReportService account statements, aged debtors). GL entries produced by AccountingService never feed back into the dealer portal because the journals do not record a dealerId. Payments cannot offset the balance, so portal totals inevitably diverge from the ledger.

Non-existent E-Bill Stage

Despite the “Billing / Tax Service” box in the diagram, there is no runtime e-bill or e-invoice implementation under erp-domain/src/main/java. The auto-approval chain stops at invoice creation; no XML/TXT generation, GST API handshake, or async dispatch jobs exist. “Sales → Invoice → Journal → E-bill → Dealer ledger” therefore halts before the statutory E-bill hop.

Assessment & Risks

  • State skew: IntegrationCoordinator.autoApproveOrder is non-transactional. If inventory reservation or journal posting succeeds but invoice persistence fails, you keep orphaned postings without compensations or idempotency guards. Replaying SalesOrderCreatedEvent will duplicate journals even though invoices are deduped.
  • Ledger mismatch: Dealer balances mutate via the denormalised column only. Reports and portals echo it verbatim, so nothing reconciles with GL data.
  • Company context fragility: Coordinator methods accept companyId but never update CompanyContextHolder; correctness hinges on the HTTP filter still being on the stack. Any async/offline trigger would post journals under whichever company happens to be in thread-local storage.
  • Accounting placeholders: postJournal auto-creates “Auto Cash/Auto Revenue” accounts and omits dealer references, so the postings are not auditable or useful for real receivables.
  • E-bill compliance gap: No e-bill generation or dispatch means regulatory workflows (GST e-invoice/e-waybill) are absent end-to-end.
