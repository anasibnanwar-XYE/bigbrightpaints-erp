# CI Debugging + Accounting Verification Plan (Codex Cloud)

## Scope and current CI entrypoint (static review)
- The repo CI job runs `mvn -B -ntp verify` from `erp-domain` and publishes Surefire + JaCoCo artifacts for debugging. Use these artifacts to inspect failing tests and coverage gaps during a Codex Cloud run. 【F:.github/workflows/ci.yml†L1-L28】

## CI failure investigation checklist (Codex Cloud)
1. **Locate the failing tests**
   - Open the `erp-domain` job logs to find the first failing test and capture the failure stack trace.
   - Download the `erp-domain/target/surefire-reports/**` artifact and identify all failing tests and any setup/teardown errors from the XML + TXT reports. 【F:.github/workflows/ci.yml†L1-L28】
2. **Map failures to suite**
   - Categorize the failing test based on suite directory (e.g., `invariants`, `regression`, `modules`, `smoke`) to determine intent and scope.
3. **Configuration cross-check**
   - Validate test-time settings (disabled schedulers, JWT secrets, etc.) match Codex Cloud runtime configuration. 【F:erp-domain/src/test/resources/application-test.yml†L1-L23】
4. **Classify failure type**
   - Deterministic logic error (assertion mismatches, invalid validation messages).
   - Nondeterministic (time/ordering, concurrency, race conditions).
   - Infra/config (missing DB, env, secrets, migrations).
   - Data/setup (seed data or fixtures diverging).
5. **Fast local reproduction**
   - If the CI failure isn’t environment-specific, run the isolated test module locally with `mvn -Dtest=<FailingTest> test`.

## Accounting verification checklist (industry-standard)
> Use these steps to verify financial integrity and spot missing logic; each item below references the current implementation.

### 1) Double-entry integrity & idempotency
- Journal entries enforce balancing (base currency) and include FX rounding tolerance with correction to largest debit/credit line when within tolerance. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountingService.java†L93-L401】
- Duplicate reference protection exists to return existing entries and prevent double postings. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountingService.java†L247-L414】

**Verification steps**
- Validate that every journal entry balances to zero after rounding adjustment; include FX/non-FX permutations.
- Confirm duplicate reference behavior is idempotent and consistent for manual vs. auto-generated references.

### 2) Period close & temporal consistency
- Journal entry posting enforces open periods when period lock is enabled; reversals require open or authorized override. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountingService.java†L256-L343】【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountingService.java†L526-L676】
- Accounting period close creates closing entries and locks periods; reopen auto-reverses closing entries. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountingPeriodService.java†L67-L207】
- Temporal balances reconstruct historical balances using accounting events. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/TemporalBalanceService.java†L23-L189】

**Verification steps**
- Validate period lock enforcement with and without admin override.
- Ensure temporal trial balance matches ledger balances for historical dates.

### 3) Chart of accounts & hierarchy
- Balance sheet and income statement hierarchies are built from account trees with total rollups. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountHierarchyService.java†L23-L150】

**Verification steps**
- Verify parent/child account type constraints and consolidated totals match expectations.

### 4) AR/AP aging + statements
- Dealer aging buckets use due-date-based bucket logic; statements compute opening balance + running balance. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AgingReportService.java†L23-L199】【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/StatementService.java†L23-L209】

**Verification steps**
- Validate bucket boundaries (0/30/60/90/90+ days) and due-date logic.
- Ensure statements reconcile to ledger totals (opening + period movements = closing).

### 5) Tax + reference integrity
- GST return sums journal lines for configured tax accounts. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/TaxService.java†L13-L70】
- Reference numbers are generated by sequence with audit trail. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/ReferenceNumberService.java†L18-L134】

**Verification steps**
- Ensure tax accounts are configured and tax totals align with invoice ledger postings.
- Confirm reference sequences are unique and auditable across modules.

### 6) Reconciliation checks
- AR/AP reconciliations compare GL balances to dealer/supplier sub-ledgers with tolerance. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/ReconciliationService.java†L26-L177】

**Verification steps**
- Ensure GL totals match sub-ledger totals within tolerance; investigate discrepancies by partner.

## Cross-module missing logic / debugging watchlist
> Items below are *potential gaps* to validate during CI debugging and manual verification.

1. **Account hierarchy totals**
   - `AccountHierarchyService` computes consolidated balances for display via recursive summation, but account nodes do not store consolidated balances. Validate UI/report usage relies on `sumNodeRecursive` totals and not raw node balances. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountHierarchyService.java†L119-L202】
2. **Aging vs. statement alignment**
   - Aging reports bucket by due date; statements track running balance by entry date. Confirm any discrepancies are expected (e.g., prepayments or credits) and documented in tests. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AgingReportService.java†L23-L199】【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/StatementService.java†L23-L209】
3. **Period enforcement bypass conditions**
   - Idempotent journal entry handling skips period validation for existing references; ensure upstream services never re-use references for different data. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountingService.java†L247-L404】
4. **Event-driven inventory journaling**
   - Inventory events skip posting when source/destination accounts are missing; ensure callers supply accounts or explicitly document manual posting requirements. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/event/InventoryAccountingEventListener.java†L90-L181】
5. **Reversal + period lock interaction**
   - Reversal checks reject locked periods and require override for closed periods. Confirm tests cover reversal vs. close/reopen flows. 【F:erp-domain/src/main/java/com/bigbrightpaints/erp/modules/accounting/service/AccountingService.java†L526-L702】
